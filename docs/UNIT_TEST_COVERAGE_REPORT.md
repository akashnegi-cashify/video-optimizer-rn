## Skipped Files with Justifications

### Category 1: Pure UI Widgets (~300+ files)

**Location:** `lib/qc/modules/*/screens/`, `lib/qc/modules/*/widgets/`, `lib/src/modules/*/widgets/`

**Justification:** These files contain `build()` methods that return widget trees. They have no business logic - only layout, styling, and widget composition. Testing them requires widget tests with `WidgetTester`, not unit tests. The Flutter framework already tests widget rendering.

**Examples:**
- `lib/qc/modules/qc_tester/calculator/screens/calculator_screen.dart`
- `lib/qc/modules/re_qc/screens/re_qc_list_screen.dart`
- `lib/src/modules/login/screens/login_screen.dart`

---

### Category 2: Generated Files (~100+ files)

**Location:** Files ending in `*.g.dart`, `*.freezed.dart`

**Justification:** Auto-generated by build_runner from annotations (`@JsonSerializable`, `@freezed`). Testing generated code is meaningless - the code generators (json_serializable, freezed) have their own test suites.

**Examples:**
- `lib/qc/modules/*/resources/*_response.g.dart`
- `lib/src/utils/media_upload/models/*_response.g.dart`

---

### Category 3: Simple DTOs (~200+ files)

**Location:** `lib/qc/modules/*/resources/*_response.dart`, `lib/qc/modules/*/models/`

**Justification:** Data Transfer Objects with only `fromJson` factory constructors that perform simple field mapping. No conditional logic, no transformations, no validation. Testing `json['field']` assignments provides no value.

**Examples:**
- `lib/qc/modules/device_details/resources/device_detail_response.dart`
- `lib/qc/modules/stock_transfer/resources/stock_transfer_response.dart`
- `lib/rms/modules/facility_list/resources/facility_list_response.dart`

**Exception:** Models with computed fields, validation, or transformation logic ARE tested (e.g., `data_wipe_filter_list_response.dart`).

### Category 5: Platform Channel Code (~5 files)

**Location:** `lib/src/channel/`

**Justification:** Platform channel code interfaces with native iOS/Android code. Unit testing requires platform-specific mocking infrastructure. These are better tested via integration tests on actual devices.

**Examples:**
- `lib/src/channel/common_method_channel.dart`
- `lib/src/channel/method_channel_names.dart`

### Category 6: Route Definitions (~15 files)

**Location:** Files matching `*_routes.dart`

**Justification:** Route files contain static route name constants and route builder functions that just instantiate widgets. No business logic to test.

**Examples:**

- `lib/src/modules/*/resources/*_routes.dart`

### Category 7: Component Registries (~10 files)

**Location:** Files matching `*_component_registry.dart`

**Justification:** Registry files map string keys to widget builders. They are configuration, not logic. The widgets they reference are tested separately.

**Examples:**
- `lib/qc/qc_component_registry.dart`
- `lib/src/app_builder/component_registry.dart`

### Category 8: Logging-Only Code (~5 files)

**Location:** `lib/src/interceptors/log_interceptor.dart`, `lib/src/libraries/logging/`

**Justification:** Code that only performs logging/printing has no testable return values. Side-effect-only code is better verified via integration testing.

**Examples:**

### Category 9: External Package Wrappers (~10 files)

**Location:** `lib/src/libraries/`

**Justification:** Thin wrappers around external packages (GetStorage, SharedPreferences, Firebase) delegate all logic to the underlying package. Testing the wrapper tests the package, not our code.

**Examples:**
- `lib/src/libraries/get_storage/app_storage.dart`
- `lib/src/libraries/shared_preferences/app_preferences.dart`
- `lib/src/libraries/firebase/firebase_helper.dart`

**Location:** `lib/rms/`, `lib/shipex/`

**Justification:** These are separate product modules with their own test suites. The current unit test initiative focuses on the QC module (`lib/qc/`) and shared services (`lib/src/`).

## Summary

This unit test suite provides comprehensive coverage of the core business logic in the flutter_trc project, focusing on:

- **Services**: HTTP request handling and API endpoint configuration
- **Providers**: State management and business logic
- **Utils**: Pure utility functions
- **Interceptors**: Request/response transformation

Files that are purely declarative (UI, config, DTOs) or auto-generated are intentionally excluded to maintain a focused, maintainable test suite.
